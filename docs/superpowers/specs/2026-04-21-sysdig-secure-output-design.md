# Sysdig Secure Output for Falcosidekick

**Date:** 2026-04-21
**Status:** Approved

---

## Summary

Add a new falcosidekick output that forwards Falco events to the Sysdig Secure Events API
(`/api/eventsDispatcher/v2/ingest`). This enables events generated by a Falco embedded
instance (e.g. from `coding-agents-kit`) to be visible in the Sysdig Secure backend
(ProdMon environment).

---

## Architecture

The output follows the standard falcosidekick HTTP-output pattern. No new dependencies
are introduced — the existing `outputs.Client` handles HTTP transport, TLS, retries,
and metrics.

### Files changed

| File | Change |
|---|---|
| `outputs/sysdigsecure.go` | New — payload types + `SysdigSecurePost()` method |
| `types/types.go` | Add `SysdigSecureOutputConfig`, `Configuration.SysdigSecure`, `Statistics.SysdigSecure` |
| `config.go` | Add default values for the `SysdigSecure` block |
| `main.go` | Initialize `sysdigSecureClient` in `init()`, gated on `APIToken != ""` |
| `handlers.go` | Dispatch `sysdigSecureClient.SysdigSecurePost()` in `forwardEvent()` |
| `stats.go` | Add `SysdigSecure: getOutputNewMap("sysdigsecure")` |
| `config_example.yaml` | Document the new `sysdigsecure` block |
| `outputs/sysdigsecure_test.go` | New — unit tests |

## Configuration

### Config struct

```go
type SysdigSecureOutputConfig struct {
    CommonConfig    `mapstructure:",squash"` // CheckCert, MutualTLS, MaxConcurrentRequests
    APIToken        string
    URL             string            // default: https://prodmon.app.sysdig.com/secure
    CustomLabels    map[string]string
    MinimumPriority string
}
```

### Activation condition

The output is enabled when `config.SysdigSecure.APIToken != ""`.

### Config file example (`config.yaml`)

```yaml
sysdigsecure:
  apitoken: ""           # Sysdig Secure API token — required to enable this output
  # url: ""              # Sysdig Secure base URL (default: https://prodmon.app.sysdig.com/secure)
  customlabels:          # optional static labels added to every event payload
    # environment: "production"
    # cluster: "my-cluster"
  minimumpriority: ""    # emergency|alert|critical|error|warning|notice|informational|debug
  checkcert: true        # verify TLS certificate (default: true)
```

### Environment variable override

All config values can be set (or overridden) via environment variables using the
`SYSDIGSECURE_` prefix:

| Variable | Description |
|---|---|
| `SYSDIGSECURE_APITOKEN` | API token (recommended for production) |
| `SYSDIGSECURE_URL` | Base URL override |
| `SYSDIGSECURE_MINIMUMPRIORITY` | Minimum priority filter |
| `SYSDIGSECURE_CHECKCERT` | TLS certificate verification |

## Payload & Data Flow

### Payload structures

```go
type sysdigSecureEvent struct {
    Timestamp    time.Time              `json:"timestamp"`
    Rule         string                 `json:"rule"`
    Priority     string                 `json:"priority"`
    Output       string                 `json:"output"`
    OutputFields map[string]interface{} `json:"output_fields"`
    Source       string                 `json:"source,omitempty"`
    Tags         []string               `json:"tags,omitempty"`
}

type sysdigSecureEventCollection struct {
    Labels map[string]string   `json:"labels"`
    Events []sysdigSecureEvent `json:"events"`
}

type sysdigSecurePayload struct {
    CloudsecEvent sysdigSecureEventCollection `json:"cloudsecEvent"`
}
```

### Field mapping

| Sysdig field | Falco field |
|---|---|
| `cloudsecEvent.events[0].timestamp` | `falcopayload.Time` |
| `cloudsecEvent.events[0].rule` | `falcopayload.Rule` |
| `cloudsecEvent.events[0].priority` | `falcopayload.Priority.String()` |
| `cloudsecEvent.events[0].output` | `falcopayload.Output` |
| `cloudsecEvent.events[0].output_fields` | `falcopayload.OutputFields` |
| `cloudsecEvent.events[0].source` | `falcopayload.Source` |
| `cloudsecEvent.events[0].tags` | `falcopayload.Tags` |
| `cloudsecEvent.labels` | `config.SysdigSecure.CustomLabels` |

### Request headers

```
Authorization:    Bearer <APIToken>
X-Sysdig-Product: SDS
Content-Type:     application/json
```

### Endpoint

```
POST <URL>/api/eventsDispatcher/v2/ingest
```

Default URL: `https://prodmon.app.sysdig.com/secure`

## Error Handling

Follows the standard falcosidekick output pattern:

- **Success:** increment `Stats.SysdigSecure OK`, Prometheus counter, OTLP metric.
- **Error:** increment `Stats.SysdigSecure Error`, log via `utils.Log`, increment error
  counters. HTTP error mapping (401, 403, 429, 500…) is handled by the existing
  `client.sendRequest()`.

## How to Configure and Run

### 1. Build falcosidekick

```bash
cd ~/work/falcosidekick
go build ./...
```

### 2. Create a config file

Create `config.yaml` in the falcosidekick directory (or any path passed to `-config`):

```yaml
sysdigsecure:
  apitoken: "<your-sysdig-secure-api-token>"
  # url: "https://prodmon.app.sysdig.com/secure"   # default, omit unless overriding
  customlabels:
    agent: "claudeCode"     # label that identifies this source
  minimumpriority: "debug"  # capture all priorities during testing
  checkcert: true
```

Or set the token via environment variable and keep the rest in the config file:

```bash
export SYSDIGSECURE_APITOKEN="<your-token>"
```

> **Note:** `customlabels` is a map type and must be set in the config file, not via
> environment variable.

### 3. Run falcosidekick

```bash
./falcosidekick -config config.yaml
```

Falcosidekick will log:

```
Enabled Outputs: [SysdigSecure]
Falcosidekick is up and listening on 0.0.0.0:2801
```

## Testing with coding-agents-kit

The `coding-agents-kit` Falco instance sends its alerts to its own embedded HTTP server
on port 2802 (`http://127.0.0.1:2802`). The simplest way to validate the full pipeline
end-to-end is to use falcosidekick's built-in `/test` endpoint, which injects a
synthetic Falco event through all enabled outputs — no Falco reconfiguration needed.

### Quick test (synthetic event via `/test` endpoint)

With falcosidekick running and configured as above:

```bash
curl -s -X POST http://localhost:2801/test
```

This triggers a synthetic `Test rule` event with priority `Debug` through all enabled
outputs, including the Sysdig Secure output. You should see the event appear in the
Sysdig Secure UI with:

- `rule`: `Test rule`
- `labels.agent`: `claudeCode`

Check falcosidekick logs for confirmation:

```
[SysdigSecure] POST OK (200)
```

### Full integration test (real coding-agents-kit event → falcosidekick → Sysdig Secure)

To route real coding-agents-kit Falco events through falcosidekick, create a user
config override that changes the `http_output` URL to falcosidekick:

> **Note:** Falco supports only one `http_output`. This redirects alerts from the
> plugin's own server (port 2802) to falcosidekick (port 2801). The coding-agents-kit
> enforcement/monitor verdict loop will not receive alerts while this override is active.

Create `~/.coding-agents-kit/config/falco.http_output_override.yaml`:

```yaml
http_output:
  enabled: true
  url: http://127.0.0.1:2801
```

Add it to the `config_files` list in `~/.coding-agents-kit/config/falco.yaml`:

```yaml
config_files:
  - ${HOME}/.coding-agents-kit/config/falco.coding_agents_plugin.yaml
  - ${HOME}/.coding-agents-kit/config/falco.http_output_override.yaml
```

Falco watches config files and will restart automatically. Now any tool call that
triggers a Falco rule will be forwarded to falcosidekick and on to Sysdig Secure,
with `labels.agent: claudeCode` attached to every event.

To restore normal coding-agents-kit operation, remove the override file from
`config_files` and Falco will restart automatically.
